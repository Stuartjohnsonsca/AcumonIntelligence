- proxy.ts exists but has an empty matcher — it does NOT protect any routes
- ALL auth checks happen server-side in page.tsx files using auth() + redirect('/login')
- This is required because edge proxy/middleware cannot read session cookies during cross-site OAuth redirect chains (Microsoft Entra ID → callback → protected page)
- DO NOT add routes back to proxy.ts matcher — it breaks MSAL login flow
- Default NextAuth cookie config is used (do NOT override with custom cookie names/SameSite)
- Protected pages: /my-account, /my-account/admin, /tools/data-extraction all check auth() themselves
- Microsoft Entra ID uses multi-tenant 'common' endpoints (NOT tenant-specific) so external users can sign in

## Parallel Extraction Pipeline Architecture
- Files are uploaded via POST /api/extraction/upload (stores in Azure inbox, creates ExtractionFile rows)
  - SHA-256 hash computed per file; duplicates detected within the same job and marked as status='duplicate' (not uploaded to blob)
  - totalFiles on ExtractionJob counts only unique files; duplicateCount tracks skipped duplicates
- POST /api/extraction/process acts as a fire-and-forget orchestrator:
  1. Calculates adaptive batch size: ≤5 files → all at once; ≤20 → 5/batch; ≤100 → 10/batch; ≤300 → 15/batch; >300 → 25/batch
  2. Fires N parallel fetch() calls to POST /api/extraction/process-batch (does NOT await them)
  3. Returns immediately with {jobId, status: "processing", totalFiles, batchCount, batchSize}
- Each batch worker (process-batch) runs in its own serverless function:
  - Processes files with concurrency limit of 2 per batch (semaphore pattern, not all-at-once)
  - Moves blobs: inbox → processing → processed (with copy-completion await before delete)
  - Calls Gemini AI for extraction, saves ExtractedRecord + category learning
  - Atomically increments processedCount/failedCount on ExtractionJob
  - When final batch completes, marks job as "complete"
- Gemini API calls have dynamic retry with exponential backoff (5 attempts, server-hint-aware delays)
  - Parses 'retry in Xs' from Gemini 429 response body and uses that as the delay (min 2s, max 60s)
  - Falls back to exponential backoff (2s, 4s, 8s, 16s) when no server hint is present
  - Adds 0-25% random jitter to prevent thundering herd across concurrent batches
  - Transient errors (429 rate limit, 500/503, timeouts) are retried; non-transient errors fail immediately
  - Error messages are stored on ExtractionFile.errorMessage for diagnostics
- Batch workers use adaptive inter-request delay: starts at 200ms, doubles on 429 errors, decays on success (min 100ms, max 15s)
- Batch orchestrator staggers batch launches by 500ms to spread initial load
- Azure blob moves await copy completion before deleting source (prevents race conditions)
- Client-side polls GET /api/extraction/status?jobId=X every 1.5s
  - Returns {total, extracted, failed, duplicated, complete} from the ExtractionJob counters
  - UI shows an animated progress bar with green (success), red (failed) segments, and duplicate count
  - Failed files show their error messages inline in the Extraction Details > Files section for diagnostics
  - Duplicates section lists skipped files with references to their originals
  - When complete=true, polling stops and full results are fetched
- Navigation-safe: once orchestrator fires batches, processing continues server-side regardless of whether user stays on the page
- File status categories: uploaded, processing, extracted, failed, duplicate

## Xero Delegated Access Flow
- Auditors do NOT log into Xero themselves — they request access from the client via email
- Flow: Auditor clicks "Collate data from Xero" → system checks if connected → if not, sends email to client contact
- **POST /api/accounting/xero/request-access** — Creates a XeroAuthRequest with a unique token (7-day expiry), sends branded email to client contact
- **GET /api/accounting/xero/request-access?clientId=X** — Returns latest request status for polling
- **Public page /xero-authorise/[token]** — Client clicks email link, sees branded page explaining what access is being granted (read-only, 30-day expiry), clicks "Connect to Xero"
- **GET /api/accounting/xero/client-connect?token=X** — Validates the delegated token, stores PKCE code_verifier in DB (not cookie — client is not authenticated), redirects to Xero OAuth
- **Xero callback** updated to handle both authenticated (direct) and delegated (token-based) flows:
  - Delegated: PKCE verifier retrieved from XeroAuthRequest DB record, no session auth required
  - On success: stores encrypted tokens, marks XeroAuthRequest as 'authorised', redirects client to confirmation page
- **XeroAuthRequest model** tracks: clientId, requestedBy, recipientEmail, token (unique), codeVerifier, status (pending/authorised/expired), expiresAt
- UI shows pending request status with "Check Status" button; once authorised, shows green confirmation

## Xero Security Compliance
- **OAuth 2.0 + PKCE (S256)**: code_verifier generated per session, stored in httpOnly cookie, sent during token exchange
- **State validation**: OAuth state with nonce stored in httpOnly cookie, validated on callback to prevent CSRF
- **Token encryption**: AES-256-GCM (exceeds Xero's AES-128 minimum) with per-encryption random IV
- **TLS 1.2+**: Enforced by Vercel (all endpoints HTTPS)
- **302 redirects**: Auth flow uses redirects, never returns tokens in response bodies
- **30-day auto-deletion**: Connections expire 30 days after authentication
  - Daily cron job (GET /api/cron/cleanup-xero at 02:00 UTC) revokes Xero connection via DELETE /connections/{tenantId} then deletes the DB row
  - Vercel Cron configured in vercel.json, authenticated via CRON_SECRET Bearer token
- **Read-only scopes**: accounting.invoices.read, accounting.payments.read, accounting.banktransactions.read, accounting.manualjournals.read, accounting.settings.read, accounting.contacts.read, accounting.attachments.read
- **No token exposure**: All Xero tokens are server-side only, never sent to the client

## Document Retention Policy
- Extraction jobs expire 121 days after creation (expires_at set on job creation)
- **40-day reminder**: email sent to the job creator when extraction is 40 days old (81 days remaining)
- **81-day reminder**: final reminder email sent at 81 days (40 days remaining)
- **121-day expiry**: blobs deleted from Azure, files marked as 'expired', job status set to 'expired'
- Daily cron job at 03:00 UTC (GET /api/cron/cleanup-extraction, secured by CRON_SECRET)
- UI shows warning banner when documents are within 30 days of expiry
- Expired jobs display "Documents expired" message instead of viewer/export options

  - Green solid dot = completed (auto-removed after 3 hours)
  - Red solid dot = error
  - Hover shows client name, activity description, and status
- **DB-backed task state**: Background tasks (Xero fetch, attachment extraction) use the `background_tasks` DB table instead of in-memory Maps. This ensures task state survives serverless instance recycling.
  - BackgroundTask model: id, userId, clientId, type ('xero-fetch' | 'xero-attachments'), status ('running' | 'completed' | 'error'), progress (JSON), result (JSON), error, timestamps
  - Indexed on (userId, status) and (createdAt) for efficient queries
- **`after()` from `next/server`**: All fire-and-forget background work uses Next.js `after()` instead of bare async IIFEs. This guarantees the serverless function stays alive until the background work completes.
  - Used in: fetch-background, fetch-attachments, and extraction process routes
- **Cleanup**: Daily cron at 02:00 UTC cleans up completed/error tasks older than 24 hours, and stuck 'running' tasks older than 2 hours
- User can navigate away and continue working while background tasks run

## Client Data Isolation
- **Shared helper**: `lib/client-access.ts` provides `verifyClientAccess(user, clientId)` and `verifyJobAccess(user, jobId)`
  - SuperAdmins can access any client
  - Other users must be in the same firm AND have a UserClientAssignment for the client
- **All extraction routes** (upload, process, status, export, document, jobs) enforce client access checks via `verifyClientAccess` or `verifyJobAccess`
- **All Xero routes** (data, status, connect, fetch-background, fetch-attachments, request-access) enforce client access checks
- **Clients API**:
  - GET /api/clients — firmId parameter only honoured for SuperAdmins; non-SuperAdmins always see their own firm's clients
  - PATCH/DELETE /api/clients/[id] — firm check ensures users can only modify clients in their own firm
  - GET /api/clients/[id]/assignments — firm check added
  - DELETE /api/clients/[id]/assignments — firm check added
- **Background tasks**: GET handlers verify task.userId matches the requesting user (or SuperAdmin)
- **Extraction jobs list**: `verifyClientAccess` check added before querying jobs for a client

## AI Cost Tracking
- Every Gemini API call (extraction + categorisation) logs token usage to the `ai_usage` table
- Token metadata captured from `result.response.usageMetadata` (promptTokenCount, candidatesTokenCount, totalTokenCount)
- Estimated cost calculated using Gemini 2.5 Flash pricing: $0.30/1M input tokens, $2.50/1M output tokens
- Records link to clientId, jobId, fileId, userId for full traceability
- **API**: GET /api/ai-usage returns firm-wide summary or per-client drill-down
- **UI**: "AI Usage" tab in My Account (visible to Firm Admin, Portfolio Owner, Super Admin)
  - Summary cards: Total AI Calls, Input Tokens, Output Tokens, Estimated Cost (USD + GBP)
  - Firm-wide "By Action" summary (e.g. Financial Data Extraction, Document Summary) with call counts and costs
  - Firm-wide "By Model" summary (e.g. meta-llama/Llama-4-Maverick) with call counts, token totals, and costs
  - Per-client table: sortable by cost, expandable rows showing action+operation breakdown, model breakdown, and recent activity log
  - Recent activity log columns: Time, Action, Model, Operation, Input tokens, Output tokens, Cost
  - Period filter: 7 days / 30 days / All time
- Cost model defined in `lib/ai-extractor.ts` via `AI_PRICING` object (easy to update when pricing changes or new models are added)
